<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Filesystem rename → git index | `git mv` preserves history; using plain `mv` would orphan history and trigger a delete+add diff that obscures the intent |
| Legacy app at `/legacy/` ↔ New app at `/` | Same origin shares localStorage (intended); separate URL scopes mean separate SW registrations (Pitfall 3) |
| Legacy SW scope after move | Implicit scope = directory of script URL = `/legacy/` (Pitfall 3); verified by Wave 6 plan 09 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | Move corrupts a v1.0 file (mid-move truncation, encoding change) | mitigate | Acceptance criteria run `wc -l public/legacy/js/app.js` >= 3000 and `grep -c 'rshape-v9' public/legacy/sw.js` == 1; `git mv` is atomic on POSIX filesystems |
| T-01-02 | Repudiation | Use of plain `mv` instead of `git mv` loses rename detection, complicates future bisects | mitigate | Acceptance criterion requires `git status --porcelain` to show `R` entries (renames), not delete+add pairs |
| T-01-03 | Denial of Service | Legacy SW at the new path 404s on its registration URL after move (a returning v1.0 user briefly sees a stale shell) | accept | Acknowledged in RESEARCH.md "Runtime State Inventory"; mitigated by browsers' graceful 404 → unregister behaviour; Wave 6 plan 09 verifies exactly one SW per scope |
| T-01-04 | Information Disclosure | None — no secrets, no env vars, no PII in the moved files | n/a | Verified by RESEARCH.md "Runtime State Inventory" `grep -rE "(api|token|secret|key)"` baseline returning only crypto constants |
</threat_model>
