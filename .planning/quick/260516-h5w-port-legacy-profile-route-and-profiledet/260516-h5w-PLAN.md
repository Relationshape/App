# Plan: Port legacy /profile route + bring ProfileDetail/ResultCard to legacy parity

## Investigation summary

**What the user reported vs what the codebase actually has:**

1. **"Profile menu button is just a dropdown" — Partial.** The current React Nav puts `ProfilePicker` (a Popover) where "Profile" would go. Clicking the trigger opens the menu but does **not** navigate anywhere. In the legacy app at `public/legacy/js/app.js#L956`, the nav-bar entry is `navLink("#/", ICONS.nav_profiles, t("nav_profiles"))` — i.e. it navigates to **`/`** (the Home/profile list view), there is **no `/profile` list route** in legacy. The user's mental model is right (clicking Profile should land on the list), but the URL is `/`, not `/profile`. The new app's `src/routes/Home.tsx` already implements the profile list view at `/`. So the fix is: make the Profile pill's label navigate to `/` while preserving the popover quick-switch.

2. **"/profile/{id} is missing styles/buttons" — Confirmed.** Concrete deltas vs legacy `viewProfile` (app.js:1564-1589) + `resultCard` (app.js:1591-1608):

   | Surface | Legacy | New React | Gap |
   |---|---|---|---|
   | Profile detail header | Edit button only | Edit + Delete | New has *extra* button (Delete on detail page). Keep — legacy has Delete only inside the Edit form, but the React behavior is a reasonable improvement. **No change needed.** |
   | Result row container | `.list-item` (avatar + body + actions grid) | `.card.result-card` (just a clickable card linking to /result/:id) | **Missing per-result action buttons entirely.** |
   | Per-result actions | Continue / View / Share / Delete (🗑) in `.li-actions` | None — the whole card is one big link to View | **4 buttons missing.** |
   | Result body line | `Updated {fmtDate(updatedAt)} · {countAnswers} answers` | `v{version}` only | **Body text mismatch.** |
   | "New map" button | `.list-add` (full-width dashed) | shadcn primary Button | **Visual mismatch on the bottom row of the list.** |
   | Home page | profile cards + imports + **templates section** | profile cards + imports (templates filtered out) | **Templates section missing.** |

3. **Imports/exports surfaced on the profile list:** Legacy `viewHome` renders `imports` and `templates` sections below profile cards. The new `Home.tsx` filters templates out at line 12 (`!isTemplateImport(i)`) — the user's "extra things like previous imported/exported data" likely refers to this templates section.

**Routing decision:** Do **not** add a new `/profile` route. The legacy app maps "Profile" to `/`, and `Home.tsx` already serves the profile-list role. Adding a separate `/profile` route would diverge from legacy and duplicate Home. Instead, make the floating-nav "Profile" pill act as a Link to `/` (the popover stays for quick-switch); this matches `navLink("#/", …)` at legacy app.js:956 exactly.

**Mid-work context honored:** Working-tree diffs on `Nav.tsx`, `ProfilePicker.tsx`, `LangToggle.tsx`, `legacy-components.css` are mid-polish (floating-pill nav, RsMenuButton extraction, v1.0 language dropdown). This plan builds on top of those — only `ProfilePicker.tsx` is touched among them, and only minimally (wrap the trigger in a Link or split into Link+Popover children). Do not revert any uncommitted work.

## Gap list (exactly what's missing from new vs legacy)

1. **Nav:** The "Profile" pill (`ProfilePicker` trigger) has no navigation behavior — only opens a popover. Add a Link-to-`/` so clicking the label navigates to the profile list (legacy parity for nav entry at `#/`).
2. **Home page:** The templates section is missing — `Home.tsx` filters all template imports out.
3. **ResultCard:** Renders as a single `<Link className="card">` instead of a `.list-item` row with a 4-button `.li-actions`.
4. **ResultCard body line:** Shows `v{version}` instead of `Updated {fmtDate(updatedAt)} · {countAnswers} answers`.
5. **ProfileDetail "new map" button:** Uses shadcn primary instead of legacy `.list-add` (full-width dashed callout).
6. **Helpers:** No `fmtDate` or `countAnswers` helpers exist in the new app — they need to be added.

## Existing primitives/conventions to follow

- `Rs*` prefix for new shared components (per user memory). We do NOT need a new Rs* here because the legacy `.li-actions` row is one-off to ResultCard. If we ever lift "icon-button-with-confirm" into a primitive, it'll be `RsIconButton` — not now.
- Use the existing `dialog()` helper (`src/lib/dialog/dialog.ts`) for the Delete confirmation, same pattern as `ProfileDetail.tsx#L30-L38`.
- i18n keys already exist for every label needed: `btn_continue`, `btn_view`, `btn_share`, `btn_delete`, `updated`, `answers`, `confirm_delete_map`, `templates_title`, `templates_sub`.
- Store actions already exist: `deleteResult` (`src/lib/storage/store.ts#L111`).
- `isTemplateImport` + `importLabel` are already exported from `src/lib/data/imports.ts` (used by `Home.tsx#L12`).

## Routing decision (recorded for the executor)

**NOT** adding `path: 'profile'` to `src/router.tsx`. The legacy `Profile` nav entry points at `#/` (Home), which the new app already serves. Making "Profile" navigate to `/` matches legacy exactly and avoids duplicating Home as a new route.

If a future task wants a dedicated `/profile` list route distinct from Home, that's a separate decision (and a divergence from legacy parity, not a parity fix).

   ```ts
   import type { Result } from '@/lib/storage/types'

   export const fmtDate = (ts: number | undefined): string =>
     ts == null ? '' : new Date(ts).toLocaleDateString(undefined, { year: 'numeric', month: 'short', day: 'numeric' })

   export function countAnswers(r: Result): number {
     let n = 0
     for (const c of Object.values(r.answers ?? {})) {
       if (typeof c !== 'object' || !c) continue
       const cat = c as Record<string, unknown>
       for (const k of Object.keys(cat)) {
         if (k === '__custom') {
           const custom = (cat.__custom ?? {}) as Record<string, { scale?: unknown } | null>
           n += Object.values(custom).filter((v) => v && (v as { scale?: unknown }).scale).length
         } else if (k === '__hidden') {
           continue
         } else {
           const val = cat[k] as { scale?: unknown } | null
           if (val?.scale) n++
         }
       }
     }
     return n
   }
   ```

   ```tsx
   import { useNavigate } from 'react-router-dom'
   import type { Result, Profile } from '@/lib/storage/types'
   import { useStore } from '@/lib/storage/store'

     const color = result.subjectColor || profile.color
     const title = (result.subject || 'Untitled') + ((result.version ?? 1) > 1 ? ` (v${result.version})` : '')
     const deleteResult = useStore((s) => s.deleteResult)

     async function onDelete() {

         title: t('confirm_delete_map'),
         body: <p>{t('confirm_delete_map')}</p>,
         actions: [
           { label: t('btn_cancel'), kind: 'ghost', value: false },

         </div>
       </div>
   ```
